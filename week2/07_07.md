### 2. 오늘 가장 의미있었던 기술적 판단

- 내가 내린 판단
    - BufferedReader의 read()도 Socket Connection에서 read() EOF가 발생하지 않을 수 있음
- 그렇게 판단한 이유
    - 기존 헤더 파싱 과정에서, readLine()역시 EOF로 readLine()의 리턴값이 null이 안나오고 입력 값을 계속 대기했던 상황이 있었음
- 고려했지만 선택하지 않은 대안
    - while loop 내에서
        
        ```java
        int readLen = bufferedReader.read(charBuff,0,contentLength-curRead);
        if (readLen==-1) break;
        ```
        
        readLen이 -1일 때 break 하는 부분을 빼려고 했음 (왜냐면 정상적인 상황에서는 전부 다 읽었을 때 0 또는 읽은 길이만큼 나오기 때문) 그러나, 만약 네트워크 문제 또는 상대가 연결을 중간에 끊어버리는 경우에는 read의 반환값이 -1이 나올 수 있으므로, 그대로 유지
        
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 만약 -1 판단을 제거 했다면, 일부 상황에서 while loop가 무한으로 반복돼 불편함 발생과 큰 성능 저하가 발생할 수 있었음
