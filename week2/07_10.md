# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라,
오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것

- 오늘 실제로 끝낸 기능/수정/학습
  - 쿠키로 전달받은 세션 ID를 기준으로 세션 존재 여부, 만료 여부, 사용자 존재 여부를 확인하는 로그인 처리 흐름을 구현했다.
  - 이 과정에서 중첩된 if문과 반복적인 Optional 확인을 줄이기 위해 Optional.filter(), flatMap(), map()을 활용하는 방법을 학습했다.


### 2. 오늘 가장 의미있었던 기술적 판단

- 내가 내린 판단
  - 값이 없을 가능성이 있다는 이유만으로 각 단계에서 Optional을 개별적으로 꺼내고 isPresent()로 확인하기보다, 세션 조회·만료 검증·사용자 조회를 하나의 흐름으로 연결하는 것이 적절하다고 판단했다.
- 그렇게 판단한 이유
  - 기존 코드는 isPresent()와 get()이 반복되고 조건문이 중첩되어, 인증 과정의 전체 흐름을 한눈에 파악하기 어려웠다. 특히 세션이 없거나 만료되었거나 사용자가 존재하지 않는 경우가 모두 로그인 실패로 이어지므로, 각 조건을 하나의 실패 가능한 처리 흐름으로 표현하는 편이 코드의 의도를 더 명확하게 보여준다고 생각했다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - filter()로 세션의 유효성을 검사하고 flatMap()으로 사용자 조회를 연결하면, 성공 경로와 실패 경로를 구분하기 쉬워진다. 이후 세션 검증 조건이 추가되더라도 인증 흐름 안에 조건을 추가할 수 있어 수정 범위도 줄일 수 있다.

### 3. 가장 막혔던 문제와 해결 과정

- 문제 상황 또는 증상
  - 클라이언트가 전달한 세션 ID를 검증하는 과정에서 세션 존재 여부, 만료 여부, 사용자 존재 여부를 각각 확인해야 했고, 그 결과 if문과 Optional.get() 호출이 여러 단계로 중첩되었다.
- 원인을 확인하기 위해 시도한 방법
  - Optional을 정말 필요한 상황에서만 사용하는 것을 고려
- 실제 원인
  - Optional 자체가 문제라기보다, Optional 값을 매번 isPresent()로 검사한 뒤 get()으로 꺼내는 방식으로 사용한 것이 중첩을 만든 원인이었다.
- 해결 방법
  - 세션 조회 결과에 filter()를 적용해 만료되지 않은 세션만 통과시키고, flatMap()으로 해당 세션의 사용자 조회 결과를 연결하는 방식으로 변경한다. 최종적으로 사용자가 존재할 때만 ifPresent() 또는 map()을 통해 로그인 상태를 처리하도록 구성한다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것
  - 여러 단계의 조회가 모두 실패할 수 있는 구조라면, 각 Optional을 개별적으로 검사하기 전에 map(), flatMap(), filter(), orElseThrow() 등으로 하나의 흐름으로 표현할 수 있는지 먼저 확인한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)

- 어떤 행동이었는지
  - 코드가 정상적으로 동작하는 것에서 끝내지 않고, 반복되는 null 검사와 깊어지는 조건문이 적절한 구조인지 의문을 갖고 다른 구현 방법을 찾아봤다.
- 그 접근이 도움이 된 이유
  - 단순히 문법 하나를 수정한 것이 아니라, Optional의 값을 직접 꺼내 검사하는 방식과 Optional 내부에서 연산을 연결하는 방식의 차이를 이해할 수 있었다. 또한 Stream에서만 사용해봤던 filter()가 Optional에도 적용될 수 있다는 점을 알게 됐다.
