# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라,
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것

* 오늘 실제로 끝낸 기능/수정/학습
* API 응답 반환 방식(ResponseEntity vs Custom Response) 논의 및 `ResponseBodyAdvice` 기반 응답 규격 통일 방안 확정



### 2. 오늘 가장 의미있었던 기술적 판단

* 내가 내린 판단
  * 컨트롤러마다 응답 객체를 감싸는 대신 Spring의 `ResponseBodyAdvice`를 활용해 전역에서 공통 응답 구조로 포장하는 방식을 채택함 (`resultType`은 boolean 타입 사용, DTO는 Java `record` 적용)


* 그렇게 판단한 이유
  * 컨트롤러마다 직접 감싸는 방식은 반복 작업이 많고 누락 시 스키마가 깨지는 문제가 발생함. `ResponseBodyAdvice`를 사용하면 프레임워크 레벨에서 스키마 통일을 강제하여 누락을 방지하고 유지보수 지점을 단일화할 수 있음


* 고려했지만 선택하지 않은 대안
  * 컨트롤러마다 직접 `CommonResponse`로 감싸서 반환하는 방식
  * `resultType`을 String("SUCCESS") 타입으로 사용하자는 대안


* 이 판단이 코드 품질/테스트/유지보수에 미친 영향
  * 컨트롤러는 도메인 객체만 반환하도록 코드가 단순해지며, 포맷 변경 시 단일 지점(`CommonResponseAdvice`)만 수정하면 됨. 프론트엔드 측에서도 Axios Interceptor를 통해 전역 에러 처리 및 파싱을 공통화할 수 있음



### 3. 가장 막혔던 문제와 해결 과정

* 문제 상황 또는 증상
  * `ResponseBodyAdvice` 적용 시 예외 응답 이중 포장 문제 및 `String` 반환 메소드에서의 `ClassCastException` 발생 가능성 확인

* 실제 원인
  * ExceptionHandler가 이미 포장한 에러 응답을 Advice가 재포장하여 SUCCESS로 덮어쓰는 문제 / `StringHttpMessageConverter`의 제네릭이 String으로 고정되어 객체 포장 시 캐스팅 에러가 발생하는 특성

* 해결 방법
  * 반환 객체(body) 타입이 이미 공통 응답 DTO인 경우 재포장하지 않도록 조건 분기 추가 / `supports()` 메소드에서 `StringHttpMessageConverter`를 적용 대상에서 제외
