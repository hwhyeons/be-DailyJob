### 2. 오늘 가장 의미있었던 기술적 판단

* 내가 내린 판단
  * [결제] 매번 카드 정보를 다시 입력해야 하는 직접 카드 결제 대신, 포인트를 선결제하고 포인트로 비용을 계산하는 방식으로 전환 (시연용 가입 3만 포인트 지급)
  * [DB] 평점(0.0~5.0) 컬럼의 tinyint 타입을 변경하지 않고 그대로 유지
  * [주소/길찾기] 매번 길찾기 API를 요청하는 대신 카카오 퀵/도보 배송 API를 활용하고, 소요시간(부르미-드리미 / 출발지-도착지)을 분리하여 DB에 저장
  * [서류 제출] 드림이 배달 시작을 위한 범죄경력회보서 제출 시 사진 한 장을 더 제출받아 확인하는 방식을 채택


* 그렇게 판단한 이유
  * [결제] KG 이니시스 등 사용 시 추가 요금 결제 때마다 카드 정보를 입력해야 하는 불편함을 줄이고 포인트를 사용하도록 유도하기 위해
  * [DB] Java 매핑 시 DB의 tinyint를 Java int로 매핑 가능하며, SQL SUM() 실행 시 반환값이 tinyint가 아니므로 오버플로우가 발생하지 않기 때문에
  * [주소/길찾기] 사용자 좌표를 가지고 매번 길찾기 API를 요청하면 비효율적이라고 판단했기 때문에
  * [서류 제출] 개인이 기관 코드를 발급받을 수는 없기 때문에


* 고려했지만 선택하지 않은 대안
  * [결제] 추가 요금 결제 시마다 카드 정보를 새로 입력받아 처리하는 방식
  * [주소/길찾기] 매번 사용자 좌표 기반으로 길찾기 API를 호출하는 방식
  * [서류 제출] 기관 코드를 발급받아 제출하는 방식



### 3. 가장 막혔던 문제와 해결 과정

* 문제 상황 또는 증상
  * 평점(0.0~5.0 점수) 평균을 구하기 위해 DB tinyint 타입 값들을 합산(sum)할 때 tinyint 표현 범위를 넘어 오버플로우가 발생할 가능성이 우려됨


* 처음 추측한 원인
  * tinyint 타입의 표현 범위 제한으로 인한 합산 시 오버플로우 발생 추측


* 원인을 확인하기 위해 시도한 방법
  * Java 매핑 방식 및 SQL SUM() 함수 사용 시 반환값 타입 확인


* 해결 방법
  * DB의 tinyint는 Java 매핑 시 int로 매핑 가능하고, SQL SUM() 실행 시 반환값이 tinyint가 아니므로 기존 tinyint 타입을 그대로 두어도 괜찮음을 확인함
