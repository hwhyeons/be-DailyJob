### 2. 오늘 가장 의미있었던 기술적 판단

* 내가 내린 판단
  * 댓글 작성 버튼 클릭 시 `postId`를 전달하기 위해, Referer 헤더를 통한 리다이렉트나 클라이언트 측 JS 파싱 대신 **서버 렌더링 시점에 직접 구체적인 URL(예: `/comment/35`)을 `<a href="...">` 태그에 바인딩하여 제공하기로 결정함**.


* 그렇게 판단한 이유
  * 이미 서버(`handleMainPage`)가 렌더링 시점에 `post.getPostId()`를 알고 있으므로, 추가적인 네트워크 왕복이나 클라이언트 사이드 연산 없이 가장 단순하고 견고하게 처리할 수 있음.
  * "댓글은 어떤 게시글의 댓글인가"라는 정보가 RequestHeader 등의 숨겨진 정보가 아닌 URL(리소스 식별자)에 명시적으로 드러나므로 REST 관점에 부합함.
  * 기존 프로젝트의 패턴(예: `/like/{postId}`)과 일관성을 유지할 수 있음.


* 고려했지만 선택하지 않은 대안
  * **대안 1 (Referer 기반 리다이렉트)**: 클라이언트가 `/comment`로 요청하면 서버가 Referer 헤더를 분석해 해당 postId로 리다이렉트하는 방식 -> 브라우저 정책/프록시 환경에 따라 헤더가 유실되거나 위조될 위험이 있고, 불필요한 302 왕복이 발생하며, 새로고침 시 postId를 잃어버리는 문제가 있어 제외함.
  * **대안 2 (JS 기반 파싱)**: 클라이언트 측 JS로 현재 URL의 PathVariable을 파싱해 `/comment/{postId}`로 직접 요청하는 방식 -> 불필요하게 JS를 실행해야 하며, 새 탭/북마크/우클릭 열기 등 브라우저 기본 내비게이션 기능이 정상 작동하지 않아 제외함.


* 이 판단이 코드 품질/테스트/유지보수에 미친 영향
  * 불필요한 리다이렉트 및 자바스크립트 의존성을 제거하여 아키텍처가 단순해지고 유지보수성이 향상됨.
  * 클라이언트와 서버 간의 불필요한 결합(Referer 헤더 추론 등)을 방지하여 코드의 동작 신뢰성을 확보함.



### 3. 오늘 잘한 행동(개발과 관련된 것)

* 어떤 행동이었는지
  * 단순히 화면을 전환하는 기능을 구현할 때 눈앞의 편리한 구현 방식에 타협하지 않고, RESTful 설계 원칙, 네트워크 비용(302 왕복), 브라우저 기본 기능(우클릭, 새 탭), 신뢰성(헤더 위조 가능성) 측면에서 각 대안의 단점과 문제점을 깊이 있게 비교하고 고민한 점.


* 그 접근이 도움이 된 이유
  * 잠재적인 버그(Referer 누락 등)와 불필요한 네트워크 비용을 사전에 차단하고, 단순하면서도 견고한 해결책을 선택할 수 있게 됨.


* 다음에도 반복하고 싶은 점
  * 설계 단계에서 "어떻게든 돌아가게 만드는 방식"에 안주하지 않고, 브라우저와 HTTP 스펙의 기본 동작 원리에 입각하여 가장 단순하고 표준에 맞는 해결책을 찾아내는 분석 태도.
